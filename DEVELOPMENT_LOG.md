**Three real bugs found in ~10 minutes of click-testing that 68 passing unit tests never caught, all fixed:**
- **`ApiExceptionHandler.handleGeneric` logged nothing.** Every unhandled 500 returned only a traceId with no server-side stack trace to look up — first login attempt returned a 500 and there was nothing in the log to explain why. Added `log.error(...)` with the exception. This is what made the next two bugs findable at all.
- **Second occurrence of yesterday's lazy-proxy bug class**: `AuthController.login` (and `/auth/me`, and everywhere else `UserQueryService.rolesOf()` is used) called `Role::getCode`/`getPermissions()` on `UserRoleAssignment.role` after the transaction closed, throwing `LazyInitializationException` — `UserRoleAssignmentRepository.findByUserId` had never been given the `JOIN FETCH` that `findByUserIdIn` (the batch form, added yesterday for the N+1 fix) already had. **Login was completely broken** - every attempt 500'd, on the very first click-test this project has ever had. Fixed by adding `JOIN FETCH a.role` to the singular query too, since its only caller always reads the role afterward. This is the same root cause as yesterday's `orgScopeNode` fix (open-in-view disabled, lazy access after the session closes) recurring in a different association - worth a note that this class of bug will keep recurring anywhere a new lazy field gets read post-transaction until there's a lint rule or convention to catch it structurally, not just as they're each individually discovered.
- **`@PreAuthorize` denials have silently 500'd since the day they were written.** Spring Security's method-security interceptor throws `AuthorizationDeniedException` from *inside* the controller method invocation (AOP), which Spring MVC's own `@RestControllerAdvice` resolution catches before the exception ever reaches `JsonAccessDeniedHandler` (which only sees denials from the filter chain, per its own comment - method-security denials never reach the filter chain at all). `ApiExceptionHandler`'s catch-all `handleGeneric` was swallowing every one of them into a generic 500. This affected all 23 existing `@PreAuthorize` checks (both yesterday's and everything from EPIC-AST) from day one - nobody had ever actually triggered a denied check end-to-end to notice. Fixed with an explicit `@ExceptionHandler(AuthorizationDeniedException.class)` returning the same 403 problem+json shape `JsonAccessDeniedHandler` uses.

**US-USR-03 (permission checks enforced at the API, not just role-name checks) - built and click-tested:**
- `CurrentUser` gained a `permissions` field (with a 3-arg compatibility constructor so the ~18 existing test call sites didn't need touching) and a `hasPermission(String)` helper (`"*"` wildcard = Super Admin's blanket grant). `JwtService` now encodes/decodes a `permissions` claim alongside `roles`. `AuthController.login` computes the same role-permission union `/me` already did and puts it on the token.
- New `PermissionChecker` bean (`@perm.has('x:y')` in SpEL), reading `CurrentUser` off the `SecurityContext`.
- **All 23 `@PreAuthorize` checks across 8 controllers converted** from `hasAnyRole('ADMIN','SUPER_ADMIN')`/`hasRole('SUPER_ADMIN')` literals to `@perm.has(...)` permission-string checks (`users:read/write`, `roles:read/write`, `org:write`, `assets:write`, `asset-categories:write`). This is the actual point of the story: a custom role created through US-USR-02 (like `WAREHOUSE_LEAD`, created and click-tested today with only `["assets:write"]`) now actually grants API access - before today it couldn't, no matter what permissions it listed, because every check only recognized a fixed set of role-name string literals.
- **Migration V18** backfills `roles:read` and `asset-categories:write` onto `ADMIN`'s permission set - required because the old role-name checks let Admin through checks the seeded permission catalog (`V15`) never actually listed. Deliberately narrow: `asset-categories:write` is *not* given to `INVENTORY_MANAGER`, preserving the existing behavior that category configuration (unlike day-to-day asset CRUD) is Administrator-only - a real distinction the flat `assets:write` permission can't express, so a second permission string was needed rather than reusing one.
- 3 new tests (`PermissionCheckerTest`: grant/deny/wildcard). **Full suite: 68/68 passing.**
- **Click-tested end-to-end**, not just unit-tested: logged in as Super Admin, Admin, an Inventory Manager, and a custom-role user; confirmed Admin's `roles:read`/`asset-categories:write` (from V18) work, confirmed Inventory Manager can write assets but is correctly **403** (not 500, post-fix) on asset-categories, and confirmed the `WAREHOUSE_LEAD` custom role - a code no `@PreAuthorize` check has ever heard of - can create an asset purely because its permission set includes `assets:write`.

**Four more real bugs found by click-testing, none caught by the (now) 88 passing unit tests:**
- **`OrgNode` had no `@PrePersist`.** Unlike every other mutable entity, it predates `BaseEntity` and was never migrated onto it - it still has its own `id`/`createdAt`/`updatedAt`, with nothing to populate them.
