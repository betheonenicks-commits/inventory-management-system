Inventory Audit Management System — Document Set Review

A joint review of the Business Requirements Document (IAMS‑BRD‑1.0), Functional Requirements Specification (IAMS‑FRS‑1.0), and Software Requirements Specification (IAMS‑SRS‑1.0), conducted across business, functional, product, architecture, security, data, and readiness lenses.

Approved w/ Major Changes
Documents: BRD v1.0 · FRS v1.0 · SRS v1.0 Source status: Draft for Review, July 2026 Review date: 6 July 2026 Findings logged: 28

Executive Summary

74/100
BRD Quality Score
Dev-Ready: Not Yet

Major Risks

  • No regulatory/compliance framework named, despite a target base spanning education, non‑profit, and corporate sectors — a sales-blocking gap for any FERPA/GDPR-adjacent deployment.
  • Acceptance criteria don't exist yet for any requirement — implementation would start against interpretation, not spec.
  • Privileged accounts (Super Administrator, Administrator) are not required to use MFA.
  • No defined bulk data migration path, despite a 90‑day, 100%‑registration business objective (BO‑1) that depends on one.
  • No accessibility or disaster-recovery objectives, despite a customer base that includes public-sector-adjacent institutions.

Major Strengths

  • Sector-agnostic, configurable data model avoids feature-fork sprawl across education/non-profit/corporate — a genuinely disciplined architectural call.
  • Non-functional requirements are concrete and numeric (100,000 assets, p95 latency, 100 concurrent users) rather than aspirational language.
  • The audit module — the system's stated core differentiator — is the most rigorously specified workflow in the set: evidence capture, immutability, sign-off, exception reporting all present.
  • Consistent ID/traceability scheme (BR‑xx / FR‑MODULE‑xx / NFR‑CATEGORY‑xx) that is genuinely friendly to both human and AI‑assisted implementation.
  • An honest, specific risk register (R‑1 to R‑5) that anticipates real adoption and scale failure modes rather than generic boilerplate.

This document set is an unusually disciplined first draft: the sector-agnostic configuration strategy, the numeric NFRs, and the audit module's completeness all read as senior product thinking. The gaps below are concentrated in three places — compliance/security posture, a handful of unfinished workflow states, and the acceptance-criteria layer that the FRS itself defers — none of which require rethinking scope, all of which should be closed before backlog creation begins in earnest.

01Business Review — Detailed Findings

Medium BO‑2, BO‑3, BO‑4 · BRD §1.3
Several business objectives lack a numeric target or baseline
Description
Success measures read as directional rather than measurable — "measurable reduction," "reduced compared to the organization's current manual process," "on demand" — with no target percentage, baseline capture step, or date.
Business impact
Leadership has no objective bar for declaring the project a success; sign-off becomes a subjective conversation rather than a scorecard.
Technical impact
Analytics and dashboard requirements can't be derived from an untargeted goal — there's nothing concrete to instrument.
Recommendation
Add a baseline-capture step (e.g., at the first full audit) and a numeric target with a time horizon to each of BO‑2, BO‑3, and BO‑4.
Suggested revision
BO‑2: "Reduce missing/unaccounted assets from the organization's measured baseline (captured at its first full audit) by ≥50% within three subsequent audit cycles."
Low BRD §1.2, §1.4.1
Deployment model (one instance per organization) is implied, not stated
Description
The document implies one on-premises instance per customer organization but never rules out multi-org hosting (e.g., a diocese with many parishes, or a district with many schools sharing one instance).
Business impact
The licensing and deployment model affects sales conversations and hierarchy design for multi-site organizations.
Technical impact
Data isolation and hierarchy-root scoping need explicit design only if multi-org-per-instance is ever intended.
Recommendation
State the model explicitly in §1.4.1.
Suggested revision
"One deployment instance serves exactly one organization, which may itself span multiple campuses/sites represented in the organizational hierarchy."

02Functional Review — Detailed Findings

High BRD §2 vs. FR‑USR‑01
The "IT / Infrastructure Team" stakeholder has no matching system role
Description
The BRD's stakeholder table names an IT/Infrastructure Team that owns deployment, backups, and LDAP/AD integration. FR‑USR‑01's default role list (Super Administrator, Administrator, Inventory Manager, Auditor, Read‑only Auditor, Department Head, Volunteer, Viewer) has no corresponding entry — it's unclear whether IT staff use Super Administrator, an undefined "System Operator" role, or no in‑app role at all.
Business impact
IT staff are either over‑provisioned into full Super Admin rights or have no defined path to the configuration screens their job requires.
Technical impact
The RBAC permission matrix can't be finalized while this mapping is unresolved.
Recommendation
Explicitly map every BRD stakeholder to exactly one default role, adding a scoped "System Operator" role if Super Administrator is too broad.
Suggested revision
FR‑USR‑01: append a stakeholder‑to‑role mapping table so every row in BRD §2 resolves to a named role.
High Risk R‑2 · NFR‑PERF‑03
No functional requirement defines bulk data import/migration
Description
Risk R‑2 promises "import tooling, validation, and a reconciliation audit," and NFR‑PERF‑03 assumes bulk import exists as an async operation, but no FR defines the mechanism itself: file format/template, field mapping, validation and error reporting, duplicate handling, or partial-failure rollback.
Business impact
Go‑live is explicitly conditioned on migrating existing spreadsheet data (BO‑1: 100% of assets registered within 90 days); an unspecified import path is a direct threat to that objective.
Technical impact
Without a defined contract, the feature gets improvised late in the build or skipped, and it has no defined interaction with FR‑AST‑10's immutable history.
Recommendation
Add an explicit import requirement: template, dry‑run validation report, partial‑success handling, and a logged reconciliation entry.
Suggested revision
New FR‑AST‑13: "The system shall support bulk asset import from a defined CSV/Excel template, producing a pre‑commit validation report and a post‑import reconciliation log entry per asset."
Medium FR‑LIF‑05, FR‑AUD‑13, FR‑LIF‑09
Reject, Archive, and Restore lifecycle states are undefined
Description
Assign, Transfer, Approve, and Dispose are well specified, but the full lifecycle expected of an enterprise system (Create/Update/Delete/Approve/Reject/Assign/Transfer/Audit/Archive/Restore/Dispose) is only partly covered. No FR states what happens when a Department Head rejects a transfer or audit, whether disposed/retired assets are archived out of active views, or whether an erroneous disposal can be reversed.
Business impact
Rejected requests risk stalling with no resubmission path; an erroneously disposed asset has no recovery path, risking permanent record loss.
Technical impact
The status state‑machine is incomplete — developers will have to invent reject/restore transitions without a spec to follow.
Recommendation
Define Reject (reason required, notify requester, return to draft) and Restore‑from‑disposal (permissioned, time‑boxed, logged as a new immutable event per FR‑AST‑10) as first‑class transitions.
Suggested revision
Add FR‑LIF‑11 (Reject workflow) and FR‑LIF‑12 (Restore‑from‑disposal, permissioned and fully logged).
Low FR‑INV‑07, FR‑RPT‑06
Vendor is referenced, not a first-class managed entity
Description
Vendor records are described as something to "maintain" and "link," but no FR defines Vendor CRUD, required fields, or deactivation of an obsolete vendor.
Business impact
Vendor reporting (FR‑RPT‑06) has no defined data‑entry path behind it.
Recommendation
Add explicit Vendor Management requirements: create/edit/deactivate, contact details, linked purchase history.

03Product Review — Detailed Findings

Medium BRD §1.4.2
No product roadmap or internal phasing beyond a single Phase 1 boundary
Description
"Phase 1" is used as the only delivery boundary; there's no MVP-within-Phase-1 breakdown, release cadence, or sequencing across the twelve functional modules.
Business impact
Stakeholders can't sequence value delivery or budget approvals; a system this broad, delivered as one release, delays first user feedback.
Technical impact
Engineering can't size a true MVP and risks over‑building before validating the audit workflow — the system's actual differentiator — with real users.
Recommendation
Split Phase 1 into an MVP (Asset + Organization + core Audit) and subsequent internal releases (Inventory, Lifecycle depth, Reporting, Dashboards), each with a target milestone.
Low SRS §5.1
UX vision is implied by two experience modes, not stated as principles
Description
An admin desktop mode and a mobile audit‑scanning mode are named, but no product‑level UX principles or design‑system commitment ties them together.
Recommendation
Add a short "Product Vision & UX Principles" subsection to the BRD (e.g., "field‑first audit UX, desk‑first admin UX, one shared design system").

04Architecture Review — Detailed Findings

High SRS §2 · NFR‑PERF‑03/04 · FR‑NTF‑04
No event-driven/asynchronous mechanism is named, despite features that require one
Description
The SRS names a layered modular monolith but never names an internal eventing mechanism, even though notify‑on‑assignment (FR‑NTF‑04), async bulk operations (NFR‑PERF‑03), and background report generation (NFR‑PERF‑04) all imply one.
Business impact
Without a defined async backbone, "on assignment, notify" style rules risk becoming slow, tightly coupled synchronous calls that don't hold up at the stated scale (100,000 assets, 100 concurrent users).
Technical impact
Retrofitting eventing after modules are wired together with direct synchronous calls is expensive, and it quietly erodes the bounded‑context isolation §2.3 otherwise establishes.
Recommendation
Name an explicit in‑process domain‑event mechanism (e.g., Spring's application event publisher backed by a transactional outbox table) for cross‑module reactions, reserving a message broker as a documented future scaling option.
Medium SRS §6 · FR‑ORG‑01
Organizational hierarchy storage strategy is unspecified
Description
A "generic, recursive tree structure" is stated, but not which representation — adjacency list, materialized path, nested set, or closure table — a choice with real query‑performance consequences at the 100,000‑asset target.
Technical impact
Adjacency‑list‑plus‑recursive‑CTE hierarchy queries degrade at scale; "all assets under this node and its descendants" is a very likely query pattern for audits and reports.
Recommendation
Specify a closure‑table or materialized‑path strategy given the stated scale and recursive‑query needs.

05Process & Workflow Review — Detailed Findings

Medium FR‑LIF‑05, FR‑AUD‑13
Approval thresholds and delegation are unconfigured
Description
"Approval where configured" (transfers) and single‑approver audit routing (Department Head) never define what triggers an approval requirement — a value threshold, cross‑department only, or something else — or who acts when that sole approver is unavailable.
Business impact
Audits and transfers can stall indefinitely if the only approver is on leave; nothing prevents a requester from also being the approver of their own request.
Recommendation
Define explicit approval‑trigger rules and a delegation/escalation path (e.g., auto‑escalate to Administrator after N days of inaction).
The core audit workflow (FR‑AUD‑01 through FR‑AUD‑19) is the most complete process in the document set: start event, scope validation, verification, exception handling, approval, certificate, and immutable history are all present. No further findings at this severity tier.

06Security Review — Detailed Findings

Critical BRD §1.2 · SRS §4.4
No regulatory/compliance requirements, despite a sector mix that clearly needs them
Description
Target sectors include schools (asset assignment can intersect with student‑linked data under FERPA in the US), churches/non‑profits (donor and volunteer personal data), and corporates in varied jurisdictions (e.g., GDPR for EU entities) — yet no requirement anywhere names a compliance framework, data‑subject rights (access/erasure), or a "compliance mode."
Business impact
A single school district or EU‑based deployment can be legally blocked from go‑live if personal‑data handling doesn't meet applicable law — this is sales‑blocking, not a nice‑to‑have.
Technical impact
Data retention, right‑to‑erasure, and consent handling for Employee/Volunteer records (FR‑ORG‑04) need to be designed into the schema now, not retrofitted after launch.
Recommendation
Add an explicit compliance section identifying which regulations apply per target sector, with matching NFRs for data‑subject access/erasure and retention limits.
High FR‑SEC‑03
MFA is "Could Have" for a system holding financial and personal data
Description
Two‑factor authentication is scoped as optional, with no differentiation by role, even though Super Administrator and Administrator accounts control the entire asset register, valuation data, and personal records of employees/volunteers.
Business impact
A compromised Super Admin account is a full breach of the very system of record BO‑1 exists to establish.
Recommendation
Promote MFA to Must Have for Super Administrator/Administrator at minimum; leave it optional for lower‑privilege roles if adoption friction is a concern.
High FR‑SEC‑01, FR‑SEC‑05
No account lockout / brute-force protection specified
Description
Password policy and session timeout are specified, but nothing addresses failed‑login throttling or lockout — a baseline enterprise control.
Recommendation
Add a requirement for configurable lockout after N failed attempts, with admin‑initiated unlock and logging of the event.
Medium FR‑SEC‑04
The activity log has no search/review interface
Description
A "complete log of user activity" is required to exist, but no requirement gives an Administrator a way to search, filter, or export it — without that, the log is a compliance artifact with no operational value.
Recommendation
Add a Security/Access Log report: filterable by user, action type, and date range, exportable per FR‑RPT‑12.

07Data Review — Detailed Findings

Medium SRS §6 · NFR‑AVAIL‑02
No data retention or archival policy
Description
Append‑only history is well specified, but no requirement states a retention duration for audit trails, backups, or disposed/retired asset records — this data grows unbounded at the 100,000+ asset target with no policy to manage it.
Recommendation
Define retention tiers, e.g. active data indefinite, backups retained 12 months, disposed‑asset records archived after 3 years of inactivity.
Low FR‑AST‑01, FR‑AST‑09
Data validation rules and data ownership are unaddressed
Description
No FR states required‑field validation for core entities, and no role is named as steward of master data quality (e.g., who is accountable when a duplicate vendor is created).
Recommendation
Add per‑entity validation rules to the eventual data dictionary, and name a data‑steward responsibility in the BRD.

08Reporting Review — Detailed Findings

Low FR‑RPT‑01 – FR‑RPT‑13
No ad hoc / self-service report builder
Description
All thirteen reports are predefined; there's no ad hoc query capability, and drill‑down isn't explicitly required alongside filtering and export.
Recommendation
Consider a Should‑Have ad hoc report builder or scoped raw‑data export for power users (Viewer/Finance persona) as a Phase 2 item.

09Dashboard Review — Detailed Findings

Low FR‑DSH‑01 – FR‑DSH‑07
Named audience dashboards (Executive/Operational/Inventory) aren't distinguished from the general Dashboard module
Description
A single configurable Dashboard module exists rather than distinct Executive, Operational, Audit, and Inventory dashboards; FR‑DSH‑07 (role/scope filtering) implies tailored views but doesn't state it.
Recommendation
Clarify in the FRS that FR‑DSH‑01–06 are configurable widgets, and that "Executive," "Operational," and "Inventory" dashboards are named widget presets rather than separate modules — a documentation clarification more than a functional gap.

10User Role Review — Detailed Findings

Medium FR‑USR‑01 – FR‑USR‑04
Role inheritance and separation-of-duties are not addressed
Description
Roles are presented flat, with no inheritance hierarchy, and no rule prevents a user from both requesting and approving their own transfer or disposal.
Recommendation
State a global separation‑of‑duties rule (a requester cannot approve their own request), and clarify whether roles are flat or inheritable by design.

11Non-Functional Review — Detailed Findings

High SRS §4.6
No accessibility requirement (WCAG / ADA / Section 508)
Description
Usability NFRs cover responsiveness and PWA installability but never mention conformance to an accessibility standard — a significant gap given schools and public‑facing non‑profits are common accessibility‑obligated procurement contexts.
Recommendation
Add: "The web application shall conform to WCAG 2.1 AA."
High NFR‑AVAIL‑01/02
No disaster-recovery objectives (RTO/RPO) or stated HA posture
Description
99.5% uptime and a "documented restore procedure" are stated, but there's no Recovery Time Objective or Recovery Point Objective, and the architecture (single DB container, single backend container) implies no redundancy — a defensible choice for small IT teams, but it should be a stated decision, not an omission.
Recommendation
Add explicit RTO/RPO targets, and state that single‑node‑with‑backup is the accepted Phase 1 DR posture, with HA documented as a future option for larger deployments.
Medium SRS §4
No localization/internationalization requirement
Description
No NFR addresses multi‑language UI, date/currency formatting, or right‑to‑left support, despite a broad target market that plausibly includes non‑English‑primary institutions.
Recommendation
Either explicitly scope English‑only for Phase 1 (a valid choice) or add an i18n‑readiness NFR — externalized strings, locale‑aware formatting — so it isn't a costly retrofit later.
Low NFR‑MAINT‑04
No monitoring/observability requirement beyond logging
Description
Structured logging with correlation IDs is specified, but nothing requires health checks, metrics, or alerting — important given small IT teams operating on‑premises without a dedicated ops function.
Recommendation
Add a health‑check endpoint and basic operational metrics NFR suitable for lightweight monitoring (e.g., checks Docker Compose can act on directly).
Low SRS §4.6, §5.1
No minimum browser support matrix
Description
"Fully responsive" is stated without naming supported browsers or versions.
Recommendation
State a support matrix, e.g. the last two versions of Chrome, Edge, Firefox, and Safari.

12AI Development Readiness — Detailed Findings

Critical FRS §4
Acceptance criteria are explicitly deferred past this document set
Description
The FRS states that Given/When/Then acceptance criteria will be written "once the product backlog is created" — meaning, as of this trio of documents, no functional requirement has testable acceptance criteria yet. This is the single largest blocker to AI‑assisted or human implementation starting directly from these documents.
Business impact
Development cannot start against a fixed, testable spec; ambiguity gets resolved ad hoc by whoever writes the code, risking rework.
Technical impact
An implementer working FR‑AST‑01 today would have to invent validation rules, error states, and required fields with no source of truth.
Recommendation
Treat this BRD/FRS/SRS trio as a strong Tier‑0 spec, but gate "start of development" on acceptance criteria, the ER diagram, and the OpenAPI contract being produced first — all three are already correctly identified as planned downstream artifacts; the gap is that none exist yet.
Low FR‑AST‑06
Custom-field mechanism is named but not enumerated
Description
"Organization‑defined custom fields... without requiring code changes" doesn't state which field types (text, number, date, dropdown, lookup) must be supported at launch.
Recommendation
Enumerate the minimum required field types before this reaches a backlog item.

13Testability Review — Detailed Findings

Medium FR‑INV‑06, FR‑LIF‑07
Several FRs describe a capability, not a measurable behavior
Description
E.g., "shall compute inventory valuation using a configurable costing method" doesn't state which methods must ship at launch, so "done" can't be verified against it.
Recommendation
Wherever an FR says "configurable," enumerate the minimum set of options required at launch so it can be tested.

14Documentation Quality — Detailed Findings

Low BRD §8
No consolidated glossary across all three documents
Description
The BRD has a glossary; the FRS and SRS introduce their own acronyms (DDD, EAV, HID, PWA, NTP) without pointing back to a shared reference.
Recommendation
Maintain one shared glossary appendix, reused or linked across all three documents.
Terminology, ID conventions (BR‑xx / FR‑MODULE‑xx / NFR‑CATEGORY‑xx), and MoSCoW prioritization are applied consistently across all three documents — a real strength for downstream traceability.

Product Improvement Recommendations

Must Have
  • Acceptance criteria for every FR before development starts.
  • Regulatory compliance section naming applicable frameworks per target sector.
  • Mandatory MFA for Super Administrator and Administrator roles.
  • Bulk import/migration specification to protect the 90-day, 100%-registration objective.
  • Reject and Restore-from-disposal lifecycle states.
Should Have
  • WCAG 2.1 AA conformance commitment.
  • DR RTO/RPO targets, with single-node-with-backup stated as the accepted Phase 1 posture.
  • Account lockout after N failed login attempts.
  • Security/Access Log report for the activity log already required to exist.
  • Vendor management module as a first-class entity.
  • Data retention policy covering audit trails, backups, and disposed-asset archival.
  • Approval delegation/escalation path for absent approvers.
Nice to Have
  • Ad hoc report builder for power users beyond the 13 predefined reports.
  • i18n-readiness (externalized strings) even if English-only at launch.
  • Health-check/monitoring endpoint for lightweight ops visibility.
  • Browser support matrix stated explicitly.
  • Consolidated glossary shared across BRD/FRS/SRS.
Future Enhancement
  • Existing roadmap items (RFID, native mobile, GIS mapping, AI-assisted analytics) remain correctly deferred.
  • ERP/General Ledger export hook for organizations needing depreciation data in a downstream finance system.
  • Webhook/event-out mechanism for third-party consumption beyond the on-premises OpenAPI surface.
